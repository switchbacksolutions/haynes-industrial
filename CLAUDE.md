## Project overview

The **Haynes Industrial** website ‚Äî a business site for an environmental services company
based in Northern & Central California specializing in industrial waste management,
recycling, demolition, and industrial cleaning. Built on Astro.js.

- **Live URL**: `https://www.haynesindustrial.com`
- **Static output** ‚Äî no server runtime; every page is pre-rendered HTML
- **Zero client JS by default** ‚Äî only the dark-mode toggle script ships JS
- **Content Collections** ‚Äî type-safe Markdown/MDX blog posts validated by Zod
- **Tailwind CSS** ‚Äî utility-first styling with a custom brand color palette
- **CloudCannon CMS** ‚Äî editors manage blog content through a visual interface
- **Netlify hosting** ‚Äî `netlify.toml` drives build, headers, and redirects

## Contact form

The contact form at `/contact` uses **Netlify Forms** for serverless submission handling.

Key attributes on the `<form>` element:
```html
<form name="contact" method="POST" data-netlify="true" netlify-honeypot="bot-field">
```

Fields:
| Name | Type | Required | Notes |
|------|------|----------|-------|
| `first-name` | text | yes | |
| `last-name` | text | no | |
| `email` | email | yes | |
| `phone` | tel | yes | |
| `project-type` | select | yes | 10 options (see below) |
| `message` | textarea | no | |
| `bot-field` | hidden | ‚Äî | Honeypot spam protection |

Project type options: Universal Waste Management, E-Waste / Electronics Recycling,
Battery Recycling, HVAC & Appliance Recycling, Medical & Lab Equipment Disposal,
Lightbulb & Lamp Recycling, Industrial Resale, Demolition Services, Industrial Cleaning, Other.

Netlify form submissions appear in the Netlify dashboard under **Forms**.
No backend code or environment variables are required.

## How to add a new blog post

1. Create a new file in `src/content/blog/` ‚Äî use kebab-case, e.g.
   `universal-waste-guide.md`.

2. Add the required frontmatter at the top:

```markdown
---
title: 'Your Post Title'
description: 'One or two sentences for SEO meta and post cards (aim for 150‚Äì160 chars).'
pubDate: '2024-06-01'          # ISO date string ‚Äî required
updatedDate: '2024-06-15'      # optional; omit if never updated
heroImage: '/images/my-image.jpg'  # optional; use an image from public/images/
tags:
  - recycling
  - compliance
draft: false                   # set true to exclude from build and RSS
---

Post body in Markdown here.
```

3. Frontmatter schema (validated by `src/content/config.ts`):

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| `title` | string | yes | Used in `<title>`, OG tags, and listings |
| `description` | string | yes | 150‚Äì160 characters recommended |
| `pubDate` | date string | yes | Coerced to `Date`; ISO 8601 format |
| `updatedDate` | date string | no | Only shown if > `pubDate` |
| `heroImage` | string (URL) | no | Relative path from `public/` or absolute URL |
| `tags` | string[] | no | Defaults to `[]` |
| `draft` | boolean | no | Defaults to `false` |

Draft posts (`draft: true`) are excluded from the blog listing, RSS feed, and sitemap.

## How to add a new page

Create an `.astro` file in `src/pages/`:

```astro
---
// src/pages/gallery.astro
import BaseLayout from '@/layouts/BaseLayout.astro';
---

<BaseLayout title="Gallery | Haynes Industrial" description="Photos of our work.">
  <main class="section-container py-16">
    <h1 class="text-4xl font-bold text-brand-800 dark:text-white">Gallery</h1>
    <!-- page content -->
  </main>
</BaseLayout>
```

The file path maps directly to the URL:
- `src/pages/gallery.astro` ‚Üí `/gallery`
- `src/pages/services/index.astro` ‚Üí `/services`

After adding a page, also add it to `NAV_LINKS` in `Header.astro` and `FOOTER_LINKS`
in `Footer.astro` if it should appear in navigation.

## Component conventions

### File naming
- Astro components: `PascalCase.astro`
- TypeScript modules: `camelCase.ts`
- Test files: `kebab-case.spec.ts` or `kebab-case.test.ts`

### Props interface
Always declare a typed `Props` interface at the top of the frontmatter block:

```astro
---
interface Props {
  title: string;
  description?: string;
  variant?: 'primary' | 'secondary';
}

const { title, description, variant = 'primary' } = Astro.props;
---
```

### Path aliases
Use the `@/` alias for imports from `src/`:

```astro
import BaseLayout from '@/layouts/BaseLayout.astro';
import FormattedDate from '@/components/FormattedDate.astro';
```

### Client-side JavaScript
Avoid client-side JS unless strictly necessary. When needed, use `<script>` tags
inside `.astro` files ‚Äî they are automatically bundled by Vite and scoped to the
component. Do NOT add `is:global` or `is:inline` unless there is a specific reason.

## Tailwind usage conventions

### Do's
- Use Tailwind utility classes for all styling.
- Use `dark:` variants for dark mode support (e.g., `dark:bg-brand-900`, `dark:text-white`).
- Use the `@layer components` block in `global.css` for repeated patterns.
- Use `class:list={[...]}` in Astro for conditional classes.

### Established component classes (global.css)

| Class | Usage |
|-------|-------|
| `.section-container` | Standard page section wrapper (max-w-5xl, responsive padding) |
| `.service-card` | Rounded card with border, shadow, hover lift effect |
| `.btn-primary` | Brand blue filled button (`bg-brand-600`, white text) |
| `.btn-secondary` | Outline button variant |
| `.tag-pill` | Rounded badge for blog tags |
| `.card` | Generic card container |
| `.prose-content` | Tailwind typography for Markdown blog content |

### Don'ts
- Do not write arbitrary inline `style` attributes unless Tailwind cannot express the value.
- Do not create new CSS files ‚Äî extend `global.css` or use `<style>` blocks.
- Do not install `clsx` or `classnames` ‚Äî Astro's `class:list` directive is sufficient.

### Dark mode
Dark mode uses the `class` strategy. Toggle by adding/removing the `dark` class on
`<html>`. The Header component handles this with a `<script>` block using `localStorage`
with system preference fallback.

### Responsive design
Always design mobile-first: base class first, then responsive overrides.
- `sm:` ‚Äî 640 px +
- `md:` ‚Äî 768 px +
- `lg:` ‚Äî 1024 px +
- `xl:` ‚Äî 1280 px +

## Testing workflow

### Unit tests (Vitest)

```bash
npm test               # run once
npm run test:watch     # watch mode
```

Place unit tests in `tests/unit/`. Test pure utility functions ‚Äî not Astro components.

### E2E tests (Playwright)

```bash
npm run test:e2e           # run all browsers
npm run test:e2e:ui        # interactive UI mode
npx playwright test --headed  # visible browser
```

Place E2E tests in `tests/e2e/`. Prefer `getByRole` locators. When adding a new page,
add a corresponding E2E spec covering: 200 status, key headings visible, nav links work,
accessibility landmarks present.

## CloudCannon CMS notes

`cloudcannon.config.yml` defines how the CMS presents content to editors.

### Collections
- **blog** ‚Üí `src/content/blog/` ‚Äî all `.md` and `.mdx` files
- **pages** ‚Üí `src/pages/` ‚Äî static pages

### Editing frontmatter
The `_inputs` block in `cloudcannon.config.yml` maps every frontmatter field to a CMS
input widget. When you add a new frontmatter field to the Zod schema in
`src/content/config.ts`, also add a corresponding entry under `_inputs` in
`cloudcannon.config.yml`.

### Upload paths
Images uploaded via the CMS are saved to `public/images/uploads/`.

```toml

  to   = "/new-url"
  status = 301
```

## Do's and Don'ts

### Do
- Keep components small and single-purpose.
- Run `npm run check` before committing ‚Äî catches type errors and Astro issues.
- Keep `draft: true` on in-progress blog posts ‚Äî they won't appear in builds.
- Use `public/images/` for all site images; prefer AVIF format (already established).
- Update `CLAUDE.md` when you change the project structure, conventions, or add tooling.

### Don't
- Don't commit `.env` ‚Äî it is in `.gitignore` for a reason.
- Don't add client-side framework components (React, Vue, etc.) without a documented reason.
- Don't import from `node_modules` paths that bypass the alias ‚Äî always use `@/` for `src/` paths.
- Don't modify `dist/` ‚Äî it is generated on every build and ignored by git.
- Don't add `console.log` statements to production code.
- Don't change `astro.config.mjs` integrations without running a full build to verify.

## Environment variables

All public variables must be prefixed `PUBLIC_` to be accessible in Astro client-side
code. Server-only variables have no prefix.

Copy `.env.example` to `.env` for local development. Never commit `.env`.
On Netlify, set variables in **Site settings ‚Üí Environment variables**.
